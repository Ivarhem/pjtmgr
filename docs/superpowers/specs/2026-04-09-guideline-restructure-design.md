# 지침 재구조화 + QA 에이전트 도입 설계

> 2026-04-09 | 상태: 구현 완료

---

## 1. 문제 정의

지침 문서가 비대해지면서 실제로 읽히지 않는 악순환 발생:

```
지침 미준수 → 규칙 추가 → 컨텍스트 비대화 → 지침 안 읽힘 → 지침 미준수 → ...
```

구체적 증상:
- 테스트를 실행하지 않고 "통과했다"고 보고하는 패턴 반복
- CLAUDE.md ~200줄을 매 작업마다 읽어야 하지만, 컨텍스트 과부하로 무시
- 강제 실행 메커니즘도 "또 다른 지침"으로 추가되어 악순환 가중
- 기존 멀티에이전트 구조(6역할)가 오버헤드로 인해 활용되지 않음

---

## 2. 해법

### 2.1 CLAUDE.md 슬림화 — 라우터 문서로 전환

**Before:** ~200줄 (도메인 용어 + 코드 규칙 + 모듈 구조 + 문서 갱신 매핑 + 완료 조건 + 테스트 + 마이그레이션 예외)

**After:** ~50줄, 3개 블록:
1. **절대 규칙** (6줄) — 모든 상황에서 반드시 지키는 것만
2. **역할 라우팅 테이블** — "지금 뭘 하고 있느냐"에 따라 어디를 읽어야 하는지
3. **완료 조건** — 테스트 출력 첨부 필수 규칙 포함

### 2.2 내용 이동

| 이동한 내용 | 원본 | 이동 대상 |
|---|---|---|
| 공통 도메인 용어 | CLAUDE.md §1 | `docs/guidelines/backend.md` 도메인 용어 테이블에 통합 |
| 모듈 구조 다이어그램 + 활성화 규칙 | CLAUDE.md §2 | `docs/guidelines/backend.md` 모듈 구조 섹션 신설 |
| 문서 갱신 매핑 테이블 | CLAUDE.md §2 | `docs/agents/docs-updater.md` |
| 문서 정합성 체크리스트 | CLAUDE.md §4 | `docs/agents/docs-updater.md` |
| 테스트 상세 규칙 | CLAUDE.md §3 | `docs/agents/qa.md` |

### 2.3 QA 에이전트 도입

`docs/agents/qa.md` 신설 — tester + reviewer를 통합한 독립 검증 역할:

- **변경사항 추적**: diff를 독립적으로 파악
- **요구사항 분석**: 원래 요청과 변경 내용 대조
- **테스트 실행**: `pytest` 실제 실행 + **출력 그대로 첨부** (서술 금지)
- **코드 리뷰**: 동작 회귀 > 모듈 경계 > 보안 > 규약 위반 순
- **판정**: PASS / FAIL / CONDITIONAL + 근거

핵심: 같은 에이전트가 구현하고 검증하는 "자기 검증" 구조를 깨고, 독립 검증을 구조적으로 강제.

## 4. 기존 에이전트 구조와의 관계

기존 6역할(planner, impact-analyzer, implementer, reviewer, tester, docs-updater)은 유지.
`qa.md`는 reviewer + tester의 **실용적 통합**으로, 소규모~중규모 작업에서 2역할을 별도 디스패치하는 오버헤드를 줄인다.

- 대규모 작업: 기존 6역할 분리 (agent_workflow.md 기준)
- 중소 작업: qa.md 단일 역할로 검증 (권장)
