# 자산 원장 / 논리 역할 뷰 계획

> 자산관리 고도화에서 `물리 자산(B뷰)`과 `논리 역할(A뷰)`을 어떻게 나눌지에 대한 1차 구현 기준 문서.
> 이 문서는 과설계를 피하면서도 교체/대체/전용/관련업체 이력을 수용하기 위한 최소 구조를 정의한다.

## 핵심 결론

- 등록과 원장은 `물리 자산(B뷰)` 기준으로 유지한다.
- 기본 조회와 운영 관점 화면은 `논리 역할(A뷰)` 기준으로 제공한다.
- 과거 정보는 덮어쓰지 않고 `역할 할당 + 이벤트 + 관련업체 이력`으로 남긴다.

즉:

```text
기록은 물리 자산
조회는 논리 역할
변경은 이벤트로 관리
```

---

## 문제 정의

아래와 같은 사례를 자산 원장에 무리 없이 담아야 한다.

1. `내부망방화벽 INT-FW(old)`가 노후화되어 `INT-FW(new)`로 교체됨
2. `INT-FW(new)` 장애로 다른 실물 장비로 대체됨
3. `INT-FW(old)`가 더 이상 운영 FW가 아니고 신규 시스템 개발용으로 전용됨

이 사례들의 공통점은 다음이다.

- 실물 장비의 식별자(`장비ID`, `serial`, `vendor/model`)는 보존되어야 한다.
- 사용자가 일상적으로 보고 싶은 것은 `내부망방화벽` 같은 운영 역할이다.
- 동일한 운영 역할을 서로 다른 시점에 서로 다른 실물이 맡을 수 있다.

따라서 `자산 1개 레코드를 계속 덮어쓰는 방식`은 장기적으로 맞지 않는다.

---

## A / B 뷰 정의

### A뷰: 논리 역할 기준

- 목록에 보이는 기준은 `내부망방화벽`, `외부망방화벽`, `개발 FW` 같은 운영 역할
- 기본적으로는 **현재(active) 상태**를 기준으로 최신 담당 자산을 보여준다
- 상세에서는 교체 이력, 이전 담당 자산, 관련 이벤트를 확인한다

### B뷰: 물리 자산 기준

- 실제 장비/인스턴스를 모두 엔티티로 등록한다
- `장비ID`, `serial`, `vendor/model`, `카탈로그`, `상태`, `관련업체`, `귀속사업`을 관리한다
- 교체/장애/전용이 발생해도 실물은 삭제하거나 덮어쓰지 않고 상태/이력으로 남긴다

## 데이터 모델 원칙

처음부터 모든 미래 요구를 다 담는 풀스키마로 가지 않는다.
대신 아래 4개 코어를 먼저 도입한다.

추가로 자산의 `유형/분류`는 고정 문자열보다 분류체계 노드 참조로 가는 것이 적절하다.
관련 세부 설계는 `docs/superpowers/plans/2026-03-26-asset-classification-scheme.md` 를 따른다.

### 1. `assets`

- 물리 자산 원장
- 기존 자산 테이블을 이 역할로 명확히 고정한다
- 실물 식별과 실물 상태를 소유한다

예:
- `asset_code`
- `project_asset_number`
- `customer_asset_number`
- `asset_name`
- `serial_no`
- `vendor`
- `model`
- `hardware_model_id`
- `status`
- `partner_id`
- `period_id`(연결 기준)

### 2. `asset_roles`

- 논리 역할/운영 엔티티
- 예: `내부망방화벽`, `외부망방화벽`, `개발 FW`, `ERP DB`

최소 필드 예시:
- `id`
- `partner_id`
- `period_id`
- `role_name`
- `role_type`
- `status`
- `note`

### 3. `asset_role_assignments`

- 어떤 물리 자산이 어떤 논리 역할을 언제 맡았는지 저장

최소 필드 예시:
- `id`
- `asset_role_id`
- `asset_id`
- `assignment_type`
- `valid_from`
- `valid_to`
- `is_current`
- `note`

### 4. `asset_events`

- 변경 액션 중심의 이벤트 로그

최소 필드 예시:
- `id`
- `asset_id`
- `asset_role_id`
- `event_type`
- `occurred_at`
- `summary`
- `details_json`
- `related_asset_id`
- `partner_id`
- `period_id`

## 관련 업체 모델

기존 `maintenance_vendor` 단일 문자열 필드는 장기적으로 부족하다.

1차 방향:
- 기존 필드는 당장 유지 가능
- 신규 구조로 `asset_related_partners`를 추가하는 것을 목표로 한다

최소 필드 예시:
- `id`
- `asset_id`
- `partner_id`
- `relation_type`
- `valid_from`
- `valid_to`
- `note`

권장 `relation_type`
- `vendor`
- `supplier`
- `maintenance`
- `installer`
- `operator`
- `rental`
- `owner`

이 구조를 쓰면 유지보수사 변경, 설치사 변경도 이력으로 남길 수 있다.

## 사례별 처리 기준

### 사례 1. 운영 FW 교체

예:
- 역할: `내부망방화벽`
- 자산: `INT-FW(old)`, `INT-FW(new)`

처리:
- `INT-FW(old)`와 `INT-FW(new)`는 각각 별도 물리 자산으로 등록
- `내부망방화벽`은 `asset_roles`의 역할 엔티티로 유지
- 역할 할당 이력으로 교체를 표현
  - old 할당 종료
  - new 할당 시작
- 이벤트 기록
  - `event_type = replacement`
  - `related_asset_id = INT-FW(old or new)`

### 사례 2. 장애 대체

기본 원칙:
- 실물이 바뀌면 새 자산으로 본다
- 기존 자산은 `failed`, `removed`, `repair`, `retired` 등 상태로 남긴다
- 새 자산을 역할에 재할당한다

예외:
- 진짜 같은 자산인데 일부 부품 교체로 `serial_no`만 바뀐 경우
- 이런 경우는 자산은 유지하고 이벤트로 `serial_update`를 남긴다

### 사례 3. 용도 전환

예:
- `INT-FW(old)`를 신규 시스템 개발용으로 전용

처리:
- 자산 자체는 유지
- 기존 역할 할당을 종료
- 새 역할(`개발 FW`)로 재할당
- 이벤트 기록
  - `event_type = repurpose`

## UI 방향

### B뷰 우선

기존 자산 목록(`/assets`)은 우선 B뷰, 즉 물리 자산 원장으로 유지한다.

이 화면에서 담당할 것:
- 자산 등록
- 카탈로그 선택
- 프로젝트 관리번호 / 고객 자산번호 관리
- 실물 정보 수정
- 관련업체 관리
- 변경 이벤트 확인

### 자산 식별자 우선순위

자산 목록과 상세에서 식별자는 아래 우선순위로 다룬다.

1. `project_asset_number`
- 프로젝트 수행 중 실무자가 가장 자주 찾는 번호
- 자산 목록의 전면 컬럼으로 사용

2. `customer_asset_number`
- 고객 자산대장 또는 고객 내부 관리번호
- 프로젝트 관리번호와 별도로 유지

3. `asset_name`
- 물리 자산 식별명
- 예: `인터넷방화벽#1(old)`, `인터넷방화벽#1(new)`
- 논리 역할명과 동일시하지 않는다

4. `asset_code`
- 시스템 내부 식별자
- 정합성과 내부 참조를 위한 코드
- 사용자 기본 화면에서는 보조 컬럼 또는 기본 숨김이 적절하다

### 자산명과 역할명 관계

- `role_name`은 논리 역할명이다
  - 예: `인터넷방화벽#1`
- `asset_name`은 물리 자산 식별명이다
  - 예: `인터넷방화벽#1(old)`, `인터넷방화벽#1(new)`
- 따라서 B뷰 자산명과 A뷰 역할명을 완전히 같은 값으로 강제하지 않는다.

대신 아래 정책을 사용한다.

- 자산 등록/수정/교체 시 `asset_name` 또는 카탈로그/프로젝트 번호를 기반으로 `role_name` 후보를 추천할 수 있다.
- 예: `인터넷방화벽#1(old)` -> `인터넷방화벽#1`
- 이 추천은 **제안 only** 이다.
- 자동 선택, 자동 생성, 자동 연결은 1차 구현에서 금지한다.
- 사용자가 직접 확인하고 수락/수정해야 한다.

### A뷰 추가

A뷰는 별도 원장 관리 화면이 아니라 `운영 역할 기준 조회 화면`으로 제공한다.

1차에서는 아래 둘 중 하나로 시작하는 것이 적절하다.

1. 자산 메뉴 안에 `역할 기준 보기` 탭 추가
2. 별도 `/asset-roles` 페이지 추가

### A뷰에서 제공할 것

- 역할명
- 현재 담당 물리 자산
- 현재 장비ID / 자산코드
- 상태
- 최근 변경일
- 상세 진입

### A뷰에서 제공하지 않을 것

- 실물 식별 정보의 직접 편집
- 자산 원장 전체 CRUD

즉, A뷰는 조회와 액션 중심이다.

## 액션 중심 UX

프론트는 모델 편집기보다 업무 액션 중심으로 가는 편이 좋다.

1차 권장 액션:

1. `교체`
- 기존 역할에 새 자산을 할당
- old 할당 종료
- replacement 이벤트 생성

2. `장애 대체`
- replacement와 유사하지만 event_type만 별도 구분

3. `용도 전환`
- 기존 역할 종료
- 새 역할 할당
- repurpose 이벤트 생성

4. `관련업체 변경`
- 유지보수사 등 관련업체 이력 추가/종료

이 액션은 자산 상세 또는 역할 상세에서 제공한다.

## 1차 구현 범위

처음부터 A/B를 완전히 분리하지 않는다.

### Phase 1

- 기존 자산 목록을 B뷰로 명확히 고정
- `asset_events` 도입
- 자산 상세에 `변경 이력` 탭 추가
- `related partners` 모델 또는 최소 유지보수사 이력 구조 도입

### Phase 2

- `asset_roles`
- `asset_role_assignments`
- A뷰(역할 기준 보기) 추가

### Phase 3

- `교체`, `장애 대체`, `용도 전환` 액션 UI
- 역할 상세 이력 타임라인

---

## 1차 완료 기준

- 물리 자산은 덮어쓰지 않고 보존된다
- 주요 변경은 이벤트로 남는다
- 관련업체 변경 이력이 남는다
- 최소 1개의 역할 기준 뷰가 존재한다
- 교체/전용 사례를 데이터 모델상 표현할 수 있다

## 한 줄 정리

`물리 자산을 원장으로 유지하고, 논리 역할은 조회/운영 뷰로 제공하며, 변화는 이벤트와 할당 이력으로 관리한다.`
