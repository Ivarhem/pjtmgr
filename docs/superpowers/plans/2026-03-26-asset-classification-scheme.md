# 자산 분류체계 설계안

> 자산 입력 편의성과 데이터 정합성을 동시에 높이기 위한 분류체계 설계 기준.
> 1차 목표는 **글로벌 기본 분류체계 + 프로젝트별 override** 를 도입하고,
> 자산/Import가 이 분류체계를 참조하도록 만드는 것이다.

## 목적

- 고객사/프로젝트마다 다른 분류 기준을 시스템이 수용할 수 있어야 한다.
- 자산 상세 입력과 Import가 같은 분류 기준을 보도록 맞춘다.
- 자유입력 분류 문자열을 줄이고, 선택형/검증형 입력으로 정합성을 높인다.
- 분류체계 설계 근거를 외부 참고 기준과 함께 설명 가능해야 한다.

## 참고 기준

이 설계는 아래 표준의 **관점**을 참고한다.

- `ISO/IEC 19770-1`
  - IT 자산관리 체계와 통제 관점 참고
  - 자산을 식별 가능하고 관리 가능한 기준 체계로 다룬다는 설명 근거
- `NIST CPE 2.3`
  - 제품 식별과 정규화 관점 참고
  - 분류체계를 직접 복제하는 것이 아니라, 제품 식별을 구조적으로 다룬다는 설명 근거

즉, 본 시스템의 분류체계는 위 표준을 직접 구현하는 것이 아니라,
`IT 자산관리 관점 + 제품 식별 관점`을 참고해 프로젝트 운영에 맞게 설계한 트리 구조다.

권장 설명 문구:

> 본 시스템의 기본 분류체계는 ISO/IEC 19770-1의 IT 자산관리 관점과
> NIST CPE 2.3의 제품 식별 관점을 참고해 설계하였으며,
> 실제 운영 분류는 프로젝트별 업무 기준에 맞게 override 가능한 트리 구조로 관리한다.

## 범위 결정

### 1차 구현 범위

- 글로벌 기본 분류체계 1개
- 프로젝트별 override 분류체계
- 자산이 분류 노드를 참조
- 자산 상세/등록에서 분류 선택형 입력
- Import에서 분류 검증 foundation
- 글로벌 기본 체계에는 일반적으로 많이 쓰는 기본 하위 분류를 함께 제공
  - 예: `하드웨어 > 네트워크 > 라우터 / L2 / L3 / L4 / 패킷브로커`
  - 예: `하드웨어 > 보안장비 > 방화벽 / VPN / IPS / DDoS`
  - 예: `하드웨어 > 서버 > x86 / UNIX / 블레이드 / GPU`
  - 예: `소프트웨어 > 운영체제 > Linux / Windows / Unix`
  - 예: `소프트웨어 > DBMS > Oracle / MS SQL / MySQL-MariaDB / PostgreSQL`
  - 예: `소프트웨어 > 미들웨어 > WEB / WAS / MQ / TP Monitor`
  - 예: `소프트웨어 > 보안소프트웨어 > DB암호화 / 시스템접근제어 / EDR-SIEM`
  - 예: `서비스/업무 > AI/모델 > LLM / 임베딩모델 / 비전모델`

### 1차에서 하지 않는 것

- 고객사별 전용 분류체계
- 외부 표준(CPE/UNSPSC) 직접 연동
- 분류체계 버전관리
- 자동 매핑/자동 추천
- 다중 프로젝트 간 실시간 참조 공유

### 현재 구현 상태 (2026-03-26)

- [x] 글로벌 기본 분류체계 / 프로젝트별 override foundation
- [x] 프로젝트 생성 시 글로벌 / 같은 고객사 기존 프로젝트 체계 복사
- [x] 프로젝트별 분류체계 관리 화면
- [x] 글로벌 기본 분류체계 관리 화면
- [x] 트리 depth 일괄 펼침, 레벨 alias 설정
- [x] 자산유형-분류 매핑 foundation
- [x] 자산목록 alias 컬럼(`대구분 / 중구분 / 소구분`)
- [x] 자산 등록/상세 수정에서 최종 분류(leaf node) 선택
- [ ] Import / Export 양식 계약
- [ ] Import 검증 규칙 연결

## 핵심 원칙

1. 시스템은 `글로벌 기본 분류체계`를 가진다.
2. 프로젝트는 필요 시 `프로젝트별 override 체계`를 가진다.
3. 자산은 `대/중/소분류 문자열`이 아니라 `최종 분류 노드`를 참조한다.
4. 프로젝트 생성 시 분류체계는 아래 셋 중 하나로 시작한다.
   - 글로벌 기본 분류체계 복사
   - 같은 고객사의 기존 프로젝트 분류체계 복사
5. 기존 프로젝트 체계를 “참조”하지 않고 “복사”한다.
   - 새 프로젝트 수정이 원본 프로젝트에 영향을 주지 않게 한다.

## 데이터 모델 초안

### 1. `classification_schemes`

- `id`
- `scope_type`
  - `global`
  - `project`
- `project_id`
  - 글로벌 체계면 `null`
- `name`
- `description`
- `source_scheme_id`
  - 복사 원본 추적용
- `is_active`
- `created_at`
- `updated_at`

### 2. `classification_nodes`

- `id`
- `scheme_id`
- `parent_id`
- `node_code`
- `node_name`
- `level`
- `sort_order`
- `is_active`
- `note`

이 구조는 다음 둘 다 수용한다.

- A 프로젝트
  - `보안 > 네트워크 보안 > 방화벽`
- B 프로젝트
  - `보안장비 > 방화벽`

즉, 화면에서는 대/중/소처럼 보일 수 있지만 실제 저장은 `트리 노드 선택`이다.

### 3. `assets`

- `classification_node_id`

자산은 최종 분류 노드 하나만 참조한다.
화면에는 필요 시 경로를 풀어서 보여준다.

### 4. `asset_type_classification_mappings`

- `id`
- `asset_type_key`
- `classification_node_code`
- `is_default`
- `is_allowed`
- `sort_order`
- `note`

이 매핑은 자산유형과 글로벌 기본 분류체계를 직접 연결한다.
중요한 점은 `classification_node_id`가 아니라 `classification_node_code` 기준으로 저장한다는 것이다.

이유:

- 프로젝트 분류체계는 글로벌 기본 체계를 복사해서 생성한다.
- 따라서 프로젝트마다 `node_id`는 달라진다.
- 반면 `node_code`는 유지할 수 있다.
- 시스템은 이 `node_code`를 기준으로 프로젝트 분류체계 안의 대응 노드를 해석할 수 있다.

예:

```text
보안 > 네트워크 보안 > 방화벽
```

## 프로젝트 생성 흐름

### 추천 정책

새 프로젝트 생성 시 분류체계 시작 방법을 선택할 수 있게 한다.

1. 글로벌 기본 체계로 시작
2. 같은 고객사의 기존 프로젝트 체계 가져오기

### 기본 추천 규칙

- 같은 고객사의 기존 프로젝트 체계가 1개면 그 체계를 기본 추천
- 여러 개면 최근 사용 프로젝트 체계를 추천
- 없으면 글로벌 기본 체계를 추천

### 중요한 구현 원칙

- 가져오기는 **복사(copy)** 다.
- 참조(link)가 아니다.

그래야 새 프로젝트에서 분류체계를 수정해도 기존 프로젝트가 흔들리지 않는다.

### foundation API

1차 foundation에서는 아래 API를 기준으로 프로젝트 생성 후속 초기화를 지원한다.

- `GET /api/v1/classification-scheme-sources?partner_id={id}`
  - 글로벌 기본 체계 목록
  - 같은 고객사의 기존 프로젝트 체계 목록
- `POST /api/v1/projects/{project_id}/classification-scheme/init`
  - `global`
  - `partner_project`
  중 하나의 mode로 새 프로젝트 분류체계를 초기화

## 입력 정책

### 자산 상세 / 자산 등록

- 분류는 자유입력이 아니라 선택형으로 바꾼다.
- 프로젝트가 선택되어 있으면 프로젝트 분류체계를 우선 사용한다.
- 프로젝트 override가 없으면 글로벌 기본 체계를 사용한다.
- 자산유형을 먼저 선택하면, 해당 자산유형에 매핑된 기본 분류와 허용 분류 범위를 참고한다.
- UI는 `최종 분류(leaf node)` 선택형을 기본으로 한다.
- 사용자는 대/중/소 전체를 강제 선택하지 않는다.
- 상위 경로는 선택된 최종 노드 기준으로 자동 표시한다.
- 원하는 분류가 없으면 자산 입력 화면에서 자동 생성하지 않고 `분류체계` 메뉴로 이동시킨다.

권장 동작:

- 자산유형별 `is_default=true` 분류는 기본 추천
- `is_allowed=true` 범위 밖의 분류는 경고 또는 차단
- 자산유형과 분류를 별개 필드로 유지하되, 입력 UX에서는 두 축이 서로 연결되도록 만든다
- 사용자 화면에서는 자산유형/분류체계/카탈로그를 별도 관리개념으로 과도하게 노출하지 않고, `카탈로그 기준관리` 안에서
  `자산유형 선택 -> 최종 분류 선택 -> 제품 연결` 흐름으로 보이게 한다.
- 최종 분류 선택기의 라벨은 현재 분류체계에서 마지막으로 설정된 alias를 사용한다.
  - 예: `소구분`, `세부유형`, `상세분류`

기본 운영 예:

- `network`
  - 기본 분류: `HW-NET`
  - 허용 분류: `HW-NET-RT`, `HW-NET-L2`, `HW-NET-L3`, `HW-NET-L4`, `HW-NET-PBR`
- `security`
  - 기본 분류: `HW-SEC`
  - 허용 분류: `HW-SEC-FW`, `HW-SEC-VPN`, `HW-SEC-IPS`, `HW-SEC-DDOS`

### Import

- Import 파일의 분류 값은 현재 프로젝트 분류체계 기준으로 검증한다.
- 다만 현재 단계에서는 **양식 계약을 먼저 확정한 뒤** 구현한다.
- 아직 결정해야 하는 항목:
  - alias 컬럼을 export에 어떻게 노출할지
  - import에서 전체 경로를 받을지, 최종 분류만 받을지
  - 미매핑/미존재 분류를 오류로 막을지, 수동 매핑 단계를 둘지

1차 구현에서는 `양식 설계 확정 -> 명시적 검증 -> 오류 또는 수동 매핑` 순서가 안전하다.

## 데이터 정합성 규칙

### 강제 규칙

- 자산은 현재 프로젝트 기준 분류체계의 active node만 선택 가능
- 프로젝트 체계 외의 노드는 저장 불가
- 비활성화된 노드는 신규 선택 불가

### 경고 규칙

- 기존 자산이 비활성 노드를 참조 중이면 경고 표시
- Import에 없는 분류 경로가 들어오면 행 단위 오류 표시

## 메뉴 배치 제안

### 글로벌 기본 분류체계

- 위치: `시스템 관리`
- 이유:
  - 시스템 공통 기준 데이터이기 때문
  - 일반 프로젝트 사용자보다 관리자 중심 기능이기 때문

### 프로젝트별 분류체계

- 위치: 메인 메뉴 레벨
- 제안:
  - 현재 `이력` 오른쪽에 신규 메뉴 `분류체계`
- 이유:
  - 프로젝트 컨텍스트를 강하게 타는 기능
  - 자산/업체/이력처럼 운영자가 자주 접근할 가능성이 높음
  - 시스템 관리 안으로 넣으면 프로젝트 단위 운영 흐름에서 너무 멀어진다

즉 메뉴 방향은 다음이 적절하다.

```text
시스템 관리 -> 글로벌 기본 분류체계
메인 메뉴 -> 프로젝트별 분류체계
```

### 시스템 관리 내 추가 배치

`시스템 관리 > 프로젝트관리` 탭은 아래 3개 섹션으로 정리하는 것이 적절하다.

1. 자산유형 관리
2. 글로벌 기본 분류체계
3. 자산유형-분류 매핑

이 배치는 아래 이유로 적절하다.

- 자산유형은 시스템 공통 기술축
- 글로벌 기본 분류체계는 프로젝트 분류의 출발점
- 자산유형-분류 매핑은 위 두 축을 연결하는 관리 규칙

즉 실제 저장 구조는 자산유형과 분류체계를 통합하지 않고 `매핑`으로 연결하는 것이 맞다.
다만 사용자에게는 이 매핑 테이블을 직접 노출하기보다, 카탈로그의 최종 분류/기본 추천 규칙으로 흡수해서 보이게 하는 편이 더 직관적이다.

권장 운영:

- 자산유형 등록/수정 시 기본 분류 노드를 함께 선택 가능
- 필요하면 `자산유형-분류 매핑` 섹션에서 추가 허용 분류 범위를 관리
- 분류 노드는 자동 생성하지 않는다
- `미할당` 임시 노드를 만들지 않는다

## 이 설계의 장점

- 대/중/소분류와 유형/세부유형 같은 서로 다른 고객사/프로젝트 구조를 하나의 트리 구조로 수용할 수 있다.
- 자유입력 분류를 줄여 정합성을 높일 수 있다.
