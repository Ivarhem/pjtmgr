# Architecture Decisions

> 아키텍처 결정 사항을 기록한다. 결정은 추가 전용(append-only)이며, 폐기 시 취소선과 이유를 남긴다.

---

## 프로젝트 기술 인벤토리 시스템으로 한정

### 이유

- 이번 시스템의 핵심 문제는 일정 관리가 아니라 기술 정보의 분산과 소실이다.
- PM Tool 기능까지 포함하면 MVP 범위가 급격히 커지고 검증 포인트가 흐려진다.

### 영향

- 태스크, 일정, 간트, 메시징, 알림은 구현 범위에서 제외한다.
- 프로젝트 화면도 진행관리보다 인벤토리 탐색과 정합성 검증에 초점을 둔다.

---

## Asset 중심 데이터 구조 채택

### 이유

- 실무에서 IP, 포트맵, 연락처, 정책 적용 현황은 결국 특정 자산과 연결되어 조회된다.
- 프로젝트 단위만으로는 실제 기술 정보 탐색 효율이 낮다.

### 영향

- `Asset`을 중심으로 `AssetIP`, `PortMap`, `AssetContact`를 연결한다.
- 프로젝트 상세 화면에서도 Asset이 핵심 진입점이 된다.

---

## 정책 정의와 적용 상태 분리

### 이유

- 정책 원본은 재사용되어야 하고, 준수 현황은 프로젝트/자산별로 달라진다.
- 향후 AI 거버넌스, 컴플라이언스, 감사 대응으로 확장하려면 구조 분리가 필수다.

### 영향

- `PolicyDefinition`과 `PolicyAssignment`를 별도 테이블로 분리한다.
- 정책 변경 이력, 예외 사유, 점검 증적을 독립적으로 관리할 수 있다.

---

## 서버 렌더링 + HTMX + AG Grid 유지

### 이유

- 데이터 밀도가 높은 업무 화면에 AG Grid가 적합하다.
- 서버 렌더링 기반 구조는 초기 복잡도를 낮추면서도 HTMX로 부분 갱신을 구현하기 쉽다.

### 영향

- FastAPI 하나에서 페이지 렌더링과 JSON API를 함께 제공한다.
- 목록/인벤토리 화면은 AG Grid, 폼/부분 업데이트는 HTMX를 우선 사용한다.

---

## PostgreSQL을 기본 DB로 채택

### 이유

- IP 주소, 고유 제약, 동시성, 마이그레이션 안정성을 고려하면 SQLite보다 PostgreSQL이 적합하다.
- 프로젝트 단위로 자산과 연결 데이터가 늘어날수록 무결성 통제가 중요해진다.

### 영향

- `DATABASE_URL` 기본 가정은 PostgreSQL이다.
- Alembic 기반 마이그레이션을 전제로 한다.
- IP 관련 타입은 PostgreSQL `inet` 활용을 우선 검토한다.

---

## 세션 기반 인증 유지

### 이유

- Jinja2 기반 서버 렌더링 구조에 가장 단순하고 자연스럽다.
- 내부 업무용 웹앱 MVP에서 JWT 기반 분산 인증 이점이 크지 않다.

### 영향

- 서버 세션을 기본 인증 수단으로 사용한다.
- API 전용 외부 클라이언트 지원은 MVP 이후 과제로 둔다.

---

## 프로젝트 범위 내 IP 중복 검증

### 이유

- 실제 운영에서 동일 프로젝트 내 IP 충돌은 치명적인 정합성 문제다.
- 전사 단위 중복 제약은 초기에는 과도할 수 있으나, 프로젝트 단위 검증은 즉시 가치가 있다.

### 영향

- `AssetIP` 저장 시 프로젝트 범위 중복 검증을 서비스 또는 DB 제약으로 수행한다.
- 향후 전사 CMDB 연동 전까지는 프로젝트 내부 정합성 보장을 우선한다.

---

## Docker Compose 기반 로컬 실행 경로 추가

### 이유

- Windows + WSL2 환경에서도 앱과 PostgreSQL 실행 절차를 일관되게 맞출 필요가 있다.
- 로컬 Python/PostgreSQL 설치 편차를 줄이면 개발 시작 비용과 환경 오염을 줄일 수 있다.

### 영향

- 기본 로컬 실행 경로로 `docker compose up --build`를 지원한다.
- 앱 컨테이너는 PostgreSQL 준비 상태를 기다린 뒤 Alembic 마이그레이션을 적용하고 서버를 시작한다.
- 비개발 환경에서는 `APP_ENV`와 `SESSION_SECRET_KEY`를 명시적으로 설정해야 한다.
- 운영값과 시크릿은 compose 파일에 직접 하드코딩하지 않고 `.env`에서 주입한다.

## IP 대역(IpSubnet)을 독립 인벤토리로 분리

### 이유

- 실무에서 IP 대역은 역할(서비스망, 관리망, 백업망, DMZ 등)에 따라 구분되며, 자산과 독립적으로 관리되어야 한다.
- 기존 AssetIP의 subnet 문자열 필드로는 대역별 역할, 지역, 상대국 등 메타데이터를 표현할 수 없다.
- 대역 단위 조회(잔여 IP, 정책 적용)를 위해 구조화된 엔티티가 필요하다.

### 영향

- `IpSubnet` 모델을 신규 추가한다 (project_id FK, name, subnet, role, vlan_id, gateway, region, floor, counterpart, description, note).
- `AssetIP`에서 `subnet` 문자열 컬럼을 제거하고 `ip_subnet_id` FK(nullable)로 대체한다.
- 대역 변경 이력은 MVP에서는 note 필드로 관리하고, 감사 로그 구현 시 구조화한다.

## 인증을 환경변수 비교에서 DB User 조회로 전환

### 이유

- 초기에는 환경변수 기반 부트스트랩 인증으로 빠르게 시작했으나, 다중 사용자 지원과 비밀번호 변경이 불가능했다.
- 사용자 관리(생성, 비활성화, 비밀번호 초기화)를 위해 DB 기반 인증이 필수다.

### 영향

- `auth/service.py`의 `authenticate()`가 DB User 테이블을 조회하도록 전환.
- `bootstrap.py`가 앱 시작 시 환경변수 기반 admin User 레코드를 자동 생성(멱등).
- 비활성 사용자(`is_active=False`)는 로그인 거부.

---

## DB 스키마 변경은 Alembic 단일 경로로 관리

### 이유

- 앱 시작 시 `create_all()`과 Alembic을 같이 쓰면 실제 스키마 기준이 둘로 갈라진다.
- 기존 로컬 DB가 이미 `create_all()`로 생성되었더라도 이후 변경은 migration 기록 기준으로 추적되어야 한다.

### 영향

- 초기 스키마를 Alembic `versions/`에 명시적으로 기록한다.
- 앱 startup은 테이블 생성 대신 DB 연결 확인만 수행한다.
- 신규/기존 로컬 DB 모두 `alembic upgrade head`를 통해 같은 revision 기준으로 맞춘다.
