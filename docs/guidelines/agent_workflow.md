# 멀티에이전트 작업 운영 지침

> 저장소에 버전관리되는 공용 작업 오케스트레이션 기준.
> 특정 CLI나 벤더 기능에 종속되지 않는다. 실제 디스패치 문법은 사용하는 도구에 맞게 번역하되, 역할·handoff·검증 규칙은 이 문서를 따른다.

---

## 목적

- 한 대화 안에서 계획, 구현, 리뷰, 테스트, 문서 갱신이 섞여 품질이 흔들리는 문제를 줄인다.
- 병렬화가 필요한 작업과 단일 책임이 필요한 작업을 분리한다.
- 로컬 ignore 경로에만 존재하는 규칙 대신, 팀이 공유 가능한 저장소 기준 문서를 둔다.

---

## 기본 원칙

- **코디네이터 1명 + 보조 역할 다수** 구조를 기본으로 한다. 사용자의 요청 해석, 범위 확정, 최종 통합 판단은 항상 코디네이터가 맡는다.
- 에이전트는 역할별로 책임을 나눈다. 한 에이전트가 계획, 구현, 리뷰, 문서 갱신을 모두 독점하지 않는다.
- 병렬화는 **서로 다른 질문** 또는 **서로 다른 write set**일 때만 허용한다.
- 멀티에이전트는 수단이다. 2파일 이하의 단순 수정, 기계적 rename, 명백한 단일 버그 수정에는 과용하지 않는다.
- 역할 카드는 도구 호출 스크립트가 아니라 **운영 규약**이다. 실제 호출 문법, 옵션명, 모델명은 사용 환경에 맞게 바꾼다.
- 참조는 섹션 제목과 파일 경로 기준으로 한다. 변동이 잦은 line number, 임시 섹션 번호, 벤더 전용 키워드에 의존하지 않는다.
- 로컬 전용 `.claude/`, `.cursor/`, 개인 홈 디렉터리 규칙만으로 팀 프로세스를 정의하지 않는다. 공용 기준은 반드시 저장소에 둔다.

---

## 역할

역할별 상세 카드와 handoff 템플릿은 `docs/agents/`에 둔다.

| 역할 | 책임 | 기본 산출물 |
| --- | --- | --- |
| planner | 요청 해석, 범위 분리, 병렬화 판단, 성공 조건 정의 | 작업 계획, 역할 배치, 위험 목록 |
| impact-analyzer | 영향 범위, 의존성, 전역 상태, 관련 문서 파악 | 영향 분석 메모, 주의사항 |
| implementer | 코드/문서 변경 실행 | 패치, 실행 결과, 잔여 리스크 |
| reviewer | 회귀/규약 위반/누락 테스트 검토 | findings, open questions |
| tester | 실행 가능한 검증 수행, 미실행 범위 기록 | 테스트 결과, 미검증 범위 |
| docs-updater | 문서 매핑 확인, guideline/structure/known issues 정리 | 갱신 대상 목록, 수정 초안 |

---

## 권장 흐름

### 1. 단일 에이전트로 충분한 경우

- 수정 범위가 작고 영향 파일이 명확하다.
- 별도 리뷰 질문이 없어도 회귀 위험이 낮다.
- 문서 갱신 대상이 자명하다.

이 경우에도 최소한 다음은 직접 확인한다.

- 영향 파일
- 테스트 또는 미실행 사유
- CLAUDE.md의 문서 갱신 매핑

### 2. 멀티에이전트를 쓰는 경우

다음 조건 중 하나 이상이면 역할 분리를 고려한다.

- 3개 이상 파일 또는 2개 이상 레이어(템플릿/JS/CSS, router/service/schema 등)를 함께 건드린다.
- 프론트엔드 전역 상태, 공용 컴포넌트, 모듈 경계, import 규칙처럼 숨은 영향 범위가 있다.
- 코드 변경과 문서 갱신 판단이 분리되어야 한다.
- 구현과 동시에 다른 질문을 병렬로 확인할 수 있다.

기본 순서는 다음과 같다.

1. planner가 범위, 성공 조건, 병렬화 가능 여부를 정리한다.
2. impact-analyzer가 영향 범위를 조사한다.
3. implementer가 변경을 수행한다.
4. reviewer와 tester가 독립적으로 검증한다.
5. docs-updater가 문서 갱신 여부를 확인한다.
6. 코디네이터가 충돌을 정리하고 최종 결과를 사용자에게 보고한다.

## handoff 규칙

각 역할에 넘길 때 최소한 아래 정보를 포함한다.

- 작업 목표
- 변경 대상 파일 또는 탐색 대상 경로
- 제외 범위
- 관련 guideline 경로
- 완료 조건
- 이미 확인된 사실과 미확인 사실

구현 역할에는 추가로 다음을 포함한다.

- write set 소유권
- 다른 에이전트가 동시에 수정 중인 파일
- 금지 사항
  - 다른 사람 변경 revert
  - 승인 없는 구조 재편
  - 사용자 요청 밖의 대규모 rename

리뷰 역할에는 추가로 다음을 포함한다.

- 구현자가 의도한 변경 요약
- 확인이 필요한 위험 가설
- 실행했거나 실행하지 못한 테스트 목록

---

## 병렬화 규칙

- 같은 파일을 동시에 수정하지 않는다.
- 하나의 질문을 여러 에이전트에게 중복으로 시키지 않는다.
- critical path를 막는 급한 분석은 코디네이터가 직접 수행한다.
- 병렬화는 다음 조합이 가장 안전하다.
  - impact-analyzer + planner 보조 조사
  - implementer + docs-updater
  - implementer + tester
  - reviewer + docs-updater

---

## 저장 경로 규칙

- 공용 기준 문서: `docs/guidelines/agent_workflow.md`
- 역할 카드: `docs/agents/*.md`
- 작업별 실행 계획: `docs/superpowers/plans/*.md`
- 로컬 벤더 설정: `.claude/`, `.cursor/`, 개인 홈 디렉터리

로컬 벤더 설정은 개인 생산성 보조 수단으로만 사용한다. 저장소 기준 규칙을 대체할 수 없다.

---

## 문서 갱신 규칙

다음 중 하나에 해당하면 이 문서를 함께 검토한다.

- 역할 추가/삭제
- handoff 형식 변경
- 병렬화 기준 변경
- 리뷰/테스트 책임 분리 방식 변경
- 특정 벤더 전용 형식 의존을 제거하거나 추가하는 경우

역할 카드만 바뀌면 `docs/agents/*.md`를 갱신한다.
워크플로 자체가 바뀌면 `CLAUDE.md`의 문서 갱신 매핑과 `docs/PROJECT_STRUCTURE.md`도 함께 본다.
