# 05. Project Rules — 프로젝트 전용 개발 지침 생성

## 목표

범용 규칙(`.ai_rules/`)을 기반으로, 이 프로젝트에 특화된 개발 지침을 생성한다.
이후 구현 단계에서 AI가 참조하는 프로젝트 전용 규칙이 된다.

## 입력

- `docs/SPEC.md` 최종본
- `docs/ARCHITECTURE.md`
- `docs/PROJECT_CONTEXT.md`
- `docs/DECISIONS.md`
- `.ai_rules/` 범용 규칙 3개
- `BOOTSTRAP_GUIDE.md`

## 출력

### 1. 프로젝트 전용 개발 지침 (예: `CLAUDE.md`, `CURSOR_RULES.md` 등)

프로젝트 루트에 생성한다.

### 2. 체크업 스킬 (`.claude/skills/checkup/skill.md`)

프로젝트 지침을 기준으로 점검·수정·지침 업데이트를 수행하되, 최초 구현 baseline 완료 여부를 먼저 확인하는 스킬을 생성한다.

```yaml
---
name: checkup
description: 최초 구현 baseline 완료 후 프로젝트 리뷰 + 코드 수정 + 지침·문서 업데이트를 수행한다
disable-model-invocation: true
---
```

스킬에 포함할 내용:

- **Phase 0 (Readiness Gate)**: 현재 프로젝트가 최초 구현 baseline 또는 MVP 완료 후의 점검/고도화 단계인지 먼저 확인
- baseline이 아직 완료되지 않았다면 스킬을 중단하고, 기존 지침을 기준으로 구현을 계속하도록 유도
- baseline 이전에는 미구현/부분구현 코드를 근거로 프로젝트 지침을 수정하지 못하게 함
- 지침이 명백히 잘못되었거나 충돌하거나 구현을 막는 경우에만 예외적으로 지침 수정 제안 허용
- **Phase 1 (리뷰)**: readiness 확인 후 지침 정합성, 아키텍처 준수, 문서 동기화, 기술 부채, 코드 품질, 보안 점검
- **Phase 2 (코드 수정)**: Phase 1 발견 사항 중 CRITICAL/WARNING을 사용자 승인 후 수정
- **Phase 3 (지침·문서 업데이트)**: baseline 완료 이후에만 반복 위반, 지침 공백, 불필요한 규칙, 지침 계층 관리, README 정리
- 각 Phase 사이에 **사용자 승인** 필수
- 지침 파일 목록을 하드코딩하지 않는다 — 메인 지침 파일의 참조 섹션을 읽어서 동적으로 파악

### 3. 부트스트랩 문서 제거

프로젝트 전용 지침 생성이 끝나면 `BOOTSTRAP_GUIDE.md`를 삭제 대상으로 명시한다.

- 이 문서는 템플릿 이해를 돕는 임시 안내서일 뿐, 프로젝트 규칙의 source of truth가 아니다
- 05단계 완료 이후에는 프로젝트 전용 지침과 `docs/` 산출물이 판단 기준이 된다
- 따라서 05단계 산출물 작성이 끝나면 `BOOTSTRAP_GUIDE.md`를 삭제하거나 최종 프로젝트 산출물에서 제외한다

## 지침 구조

범용 규칙을 **상속**하고, 프로젝트 맥락에서 **구체화**한다.

### 1. 도메인 용어 정의

- SPEC의 Domain Model에서 추출
- 코드에서 사용할 표준 용어 확정 (DB명, API명, UI 표시명)
- 동의어 사용 금지 목록

### 2. 코드 규칙

- ARCHITECTURE에서 확정된 기술 스택 기반
- 언어별 포매터, 린터, 타입 체크 설정
- 레이어별 코드 패턴 (예: 라우터는 얇게, 서비스에 로직 집중)

### 3. 명명 규칙

- 파일명 패턴 (레이어별)
- 함수명 패턴 (CRUD, 비-CRUD)
- API 경로 패턴
- HTML/CSS 명명 패턴

### 4. 아키텍처 규칙

- `.ai_rules/AI_ARCHITECTURE_GUARD.md`를 프로젝트에 맞게 구체화
- 비즈니스 로직 위치, 예외 처리 패턴, 데이터 접근 패턴

### 5. 데이터 원칙

- 금액/날짜 등 도메인 특화 데이터 처리 규칙
- 유효성 검사 패턴

### 6. UI/UX 원칙

- 언어, 날짜/숫자 표시 형식
- 컴포넌트 라이브러리 사용 규칙
- 웹앱인 경우 스타일 일관성 규칙을 반드시 포함한다
- 공통 스타일 토큰, 공용 컴포넌트, 페이지 전용 스타일의 책임 범위를 분리한다
- 템플릿 구조 변경 시 JS 셀렉터/이벤트 바인딩 영향 점검 규칙을 포함한다
- 반응형, 라이트/다크 테마, 권한별 UI 노출에 대한 최소 검증 체크리스트를 포함한다
- 외부 CDN 기반 UI 자산 사용 기준과 로컬 자산 우선 원칙을 포함한다

### 7. 보안 / 인증 / 권한

- 인증 방식, 권한 모델
- 보안 제약

### 8. 문서 갱신 규칙과 완료 조건

#### 변경-문서 매핑 표

`AI_DEV_GLOBAL_RULES.md` §6의 기본 매핑을 프로젝트에 맞게 구체화하여 지침에 포함한다.
프로젝트 지침에는 사실 문서의 즉시 갱신과, MVP baseline 이후에만 수행하는 지침 보강을 구분하는 원칙을 포함한다.
예시:

| 변경 유형 | 갱신 대상 |
| --------- | --------- |
| 비즈니스 규칙 변경 | MVP baseline 이후 프로젝트 지침 해당 섹션 |
| 권한 변경 | 인증/권한 지침 |
| 프론트엔드 패턴 변경 | 프론트엔드 지침 |
| 아키텍처 결정 | `docs/DECISIONS.md` (추가 전용) |
| 임시 우회/제약 추가 | `docs/KNOWN_ISSUES.md` |
| 임시 우회 해소 | `docs/KNOWN_ISSUES.md` (항목 삭제) |
| 모델/API/파일 구조 변경 | 문서 갱신 불필요 (코드가 source of truth) |

#### 완료 조건 (Definition of Done)

코드 변경이 "완료"되려면 다음을 모두 충족해야 한다:

1. 코드 변경 완료
2. 관련 테스트 통과 (새 기능은 테스트 추가)
3. 매핑 표에 해당하는 사실 문서가 있으면 갱신 완료
4. MVP baseline 이후의 작업이라면, 매핑 표에 해당하는 프로젝트 지침 보강 여부까지 확인 완료
5. 해결된 KNOWN_ISSUES 항목이 있으면 삭제 완료

### 9. Source of Truth 원칙

프로젝트 지침 또는 README에 아래 안내를 포함하여
하나의 사실을 여러 문서가 중복 소유하지 않도록 한다.

| 정보 유형 | Source of Truth | 문서에 쓰면 안 되는 것 |
| --------- | -------------- | -------------------- |
| 모델 필드, API 엔드포인트 | 소스 코드 | 전체 필드/엔드포인트 나열 |
| 비즈니스 규칙, 코딩 규칙 | 프로젝트 지침 | — |
| 아키텍처 결정 배경 | `docs/DECISIONS.md` | 다른 문서에 산발적 설명 |
| 알려진 제약/임시 우회 | `docs/KNOWN_ISSUES.md` | README에 중복 기술 |
| 구현 현황, 향후 계획 | 이슈 트래커 또는 `docs/TODO.md` | 프로젝트 지침에 스냅샷 |

## 원칙

- 범용 규칙과 **충돌하지 않는다** — 범용 규칙은 항상 유효하고, 전용 지침은 구체적 적용 방법을 정의
- **코드에서 실제로 검증 가능한 규칙**만 작성한다 (예: "깔끔하게 작성" 같은 주관적 규칙 금지)
- 프로젝트 진행에 따라 `06_review` → `07_guideline_update`로 지속 개선하되, 이는 최초 구현 baseline 완료 후의 피드백 루프다
- `BOOTSTRAP_GUIDE.md`는 템플릿 온보딩용 임시 문서이며, 05단계 완료 후 제거한다
- **지침이 길어지면 계층화한다** — 모든 작업에 필요한 핵심 규칙만 메인 파일에 유지하고,
  특정 작업에만 필요한 규칙(프론트엔드, 인증/권한, Excel 등)은 별도 파일로 분리하여 필요 시 참조하도록 한다.
  메인 파일 상단에 하위 지침 파일 목록을 정의하여 **단일 원본(Single Source of Truth)**을 유지한다
- 웹앱 프로젝트라면 "스타일 일관성"을 추상 문구로 두지 말고, 재사용 우선순위, 금지 패턴, 수정 후 확인 항목까지 구체적으로 적는다
