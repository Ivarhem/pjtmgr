# 06. Review — 프로젝트 점검

## 목표

프로젝트 전체를 검토하여 구조 안정성과 지침 준수 여부를 확인한다.
코드 리뷰가 아니라 **프로젝트 건강 상태 점검**이다.

## 입력

- 프로젝트 전체 코드
- 프로젝트 전용 개발 지침 (05단계 산출물)
- `.ai_rules/` 범용 규칙
- `docs/ARCHITECTURE.md`
- `docs/DECISIONS.md`

## 점검 항목

### 1. 지침 정합성

- 코드가 프로젝트 전용 지침을 따르는가
- 명명 규칙, API 경로 패턴, 레이어 규칙 위반 여부
- 도메인 용어 통일 여부

### 2. 아키텍처 준수

- `.ai_rules/AI_ARCHITECTURE_GUARD.md` 기준 위반 여부
- 비즈니스 로직이 올바른 레이어에 있는가
- 순환 의존성 존재 여부
- 단일 책임 원칙 준수 여부

### 3. 문서 동기화

- `docs/KNOWN_ISSUES.md`에 이미 해결된 항목이 남아있는가
- 프로젝트 지침의 비즈니스 규칙/코딩 규칙이 실제 코드 패턴과 일치하는가
- 코드가 source of truth인 정보(모델 필드, API 상세)가 문서에 중복 나열되어 부패하고 있는가
- `docs/DECISIONS.md`에 기록되지 않은 암묵적 아키텍처 결정이 있는가
- 프로젝트 구조(파일 트리)가 README와 일치하는가
- 변경-문서 매핑 표(프로젝트 지침)가 실제 변경 패턴을 커버하는가

### 4. 기술 부채

- TODO/FIXME/HACK 주석
- 임시 구현이 남아있는 곳
- 테스트가 필요한데 없는 곳

### 5. 확장 리스크

- 현재 구조에서 향후 기능 추가 시 대규모 수정이 필요한 곳
- 하드코딩된 값 (설정으로 빼야 할 것)

## 출력

각 발견 사항을 다음 4단계로 분류한다.

| 등급 | 의미 | 기준 |
|------|------|------|
| 🔴 반드시 수정 | 동작 오류 또는 구조 붕괴 위험 | 방치 시 다른 작업에 영향 |
| 🟡 권장 개선 | 지침 위반 또는 기술 부채 | 지금 수정이 가장 비용 적음 |
| 🟢 유지 가능 | 개선 여지는 있으나 현재 문제 없음 | 다른 작업 중 기회되면 수정 |
| ⚪ 향후 검토 | 확장 시 고려 필요 | 지금은 조치 불필요 |

## 주의사항

- 발견 사항마다 **파일 경로와 해당 지침 조항**을 명시한다
- "좋은 코드" 같은 주관적 평가는 제외한다
- 개선안은 현재 구조 내에서 가능한 것만 제안한다
