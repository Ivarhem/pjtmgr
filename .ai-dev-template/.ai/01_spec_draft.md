# 01. Spec Draft — 명세 초안 작성

## 목표

사용자의 아이디어를 구조화된 개발 명세 초안으로 변환한다.
코드를 작성하는 것이 아니라, 문제를 명확히 하고 구현 가능한 spec을 만드는 것이다.

## 입력

- 사용자의 아이디어 설명 (구두, 메모, 기존 문서 등)
- 기존 시스템이 있다면 현재 상태

## 출력 — `docs/SPEC.md` 초안

다음 구조로 작성한다.

### 1. Problem Definition

- 누구의, 어떤 문제를 해결하는가
- 현재 어떻게 해결하고 있고, 왜 불충분한가

### 2. Users

- 사용자 유형과 각 유형의 규모
- 각 사용자의 핵심 목표

### 3. Scope

- **Must** — 없으면 사용할 수 없는 기능
- **Optional** — 있으면 좋지만 없어도 동작하는 기능
- **Out of Scope** — 명시적으로 제외하는 것

### 4. Domain Model

- 핵심 엔티티와 관계 (텍스트 또는 다이어그램)
- 주요 용어 정의

### 5. Functional Requirements

- 기능 단위로 정리
- 각 기능의 입력, 처리, 출력 명시

### 6. Non-functional Constraints

- 기술 스택 제약
- 성능, 보안, 접근성 요구사항
- 운영 환경 제약

### 7. Risks

- 기술적 리스크
- 범위 리스크
- 의존성 리스크

## 주의사항

- 구현 방법이 아닌 **무엇을** 만들지에 집중한다
- 모호한 요구사항은 추측하지 말고 `[확인 필요]`로 표시한다
- 이 단계에서 기술 스택을 확정하지 않는다 (제약이 있으면 Constraints에 기록)
