# 07. Guideline Update — 지침 개선

## 목표

리뷰 결과와 구현 경험을 기반으로 개발 지침을 개선한다.
지침이 코드와 함께 진화하는 피드백 루프를 형성한다.

## 입력

- 06단계 리뷰 결과
- 구현 과정에서 발생한 문제 이력
- 현재 프로젝트 전용 개발 지침
- `docs/DECISIONS.md`

## 분석 항목

### 1. 반복 위반

같은 지침을 여러 곳에서 위반했다면:

- 규칙이 모호한가 → **규칙을 구체화**한다
- 규칙을 찾기 어려운가 → **위치를 재배치**한다
- 규칙이 비현실적인가 → **규칙을 수정**한다

### 2. 지침 공백

지침에 없는데 일관성이 필요했던 패턴:

- 코드에서 이미 관행이 형성되었는가 → **문서화**한다
- 아직 관행이 없는가 → **규칙을 신설**한다

웹앱 프로젝트에서는 특히 다음 공백을 우선 점검한다.

- 공통 스타일 토큰 없이 화면별로 색상/간격/반경이 분산되었는가
- 템플릿 수정 후 JS 셀렉터 또는 이벤트 바인딩이 깨졌는가
- 라이트/다크 테마 또는 모바일 레이아웃 검증 규칙이 빠져 있는가
- 외부 CDN 자산 사용 기준이 없어 런타임 의존성이 불안정한가

### 3. 불필요한 규칙

다음에 해당하면 제거를 검토한다:

- 한 번도 위반된 적 없고, 위반 가능성도 낮은 규칙
- 다른 규칙과 중복되는 규칙
- 프로젝트 초기에만 유효했고 현재는 무의미한 규칙

### 4. 지침 계층 관리

지침이 계층화되어 있는 경우:

- 메인 파일(항상 로드)에 특정 작업에만 필요한 규칙이 있으면 → 하위 지침 파일로 이동
- 하위 지침 파일에 모든 작업에 필요한 규칙이 있으면 → 메인 파일로 승격
- 메인 파일과 하위 지침 간 규칙 중복·충돌 해소
- 메인 파일의 하위 지침 참조 목록이 실제 파일과 일치하는지 확인

### 5. 새로운 결정

구현 과정에서 내린 설계 결정 중 `docs/DECISIONS.md`에 기록되지 않은 것.

### 6. 변경-문서 매핑 표 검증

- 매핑 표에 정의된 변경 유형이 실제 발생하는 변경을 커버하는가
- 매핑에 없는 변경이 문서 부패를 유발했는가 → 매핑 항목 추가
- 완료 조건(DoD)이 지켜지고 있는가 (KNOWN_ISSUES.md에 해결 항목 방치 등)
- source of truth 원칙이 지켜지는가 (코드가 source of truth인 정보가 문서에 중복 나열되지 않는가)

## 출력

### 지침 변경 제안

각 제안에 다음을 포함한다.

```
## [추가/수정/제거]: [규칙 제목]

현재: [현재 규칙 또는 "없음"]
제안: [변경 내용]
이유: [왜 변경이 필요한가 — 구체적 사례 포함]
```

### DECISIONS.md 추가 항목

기록되지 않은 설계 결정 목록.

## 원칙

- 지침은 **가능한 한 단순하게** 유지한다 — 규칙 수가 늘면 읽지 않게 된다
- 하나의 규칙으로 합칠 수 있는 것은 합친다
- 규칙을 추가할 때 기존 규칙과 충돌하지 않는지 확인한다
- 범용 규칙(`.ai_rules/`)은 이 단계에서 수정하지 않는다 — 프로젝트 전용 지침만 수정
