# 04. Architecture — 아키텍처 설계

## 목표

최종 spec을 기반으로 프로젝트 아키텍처를 설계한다.
구조는 단순하지만 확장을 막지 않는 형태여야 한다.

## 입력

- `docs/SPEC.md` 최종본 (03단계 산출물)
- `.ai_rules/` 범용 규칙

## 출력 — `docs/ARCHITECTURE.md`

다음을 포함한다.

### 1. 기술 스택

- 언어, 프레임워크, 주요 라이브러리
- 선택 이유 (간략히)

### 2. 프로젝트 폴더 구조

- 디렉토리 트리와 각 디렉토리의 역할
- 파일 명명 규칙

### 3. 핵심 모듈 역할

- 각 모듈/레이어의 책임 정의
- 모듈 간 의존 방향

### 4. 데이터 흐름

- 사용자 요청 → 응답의 전체 흐름
- 주요 데이터 변환 지점

### 5. 데이터 모델

- 테이블/엔티티 구조
- 핵심 관계

### 6. 확장 포인트

- 향후 기능이 추가될 위치 (구현하지 않되 자리를 명시)
- 현재 구조에서 변경 없이 확장 가능한 부분

## 주의사항

- `.ai_rules/AI_ARCHITECTURE_GUARD.md`의 책임 분리 원칙을 준수한다
- `.ai_rules/AI_EXTENSIBILITY_BALANCE.md`의 "뼈대는 지금, 살은 나중" 원칙을 따른다
- 과도한 추상화를 피한다 — 사용처가 1개뿐인 인터페이스/팩토리는 만들지 않는다
- 기술 스택 선택의 이유를 `docs/DECISIONS.md`에 기록한다
