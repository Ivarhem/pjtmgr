# Extensibility Balance Rules

목표: 과설계를 피하면서 확장을 막지 않는 구조 유지

---

## 1. 뼈대는 지금, 살은 나중

구현 우선순위(GLOBAL_RULES §2)에 따라 단계별로 구현한다.

초기 — 핵심 기능 동작:

- 책임 분리 (레이어 구조)
- 핵심 데이터 모델
- 공통 진입점 (라우터, 설정)
- 기본 CRUD
- 도메인 무결성 검증과 입력 정규화
  - 필수 필드 검증
  - 상태 전이 규칙 검증
  - 날짜/금액/식별자 canonical format 정규화

그 다음 — 사용자 편의성:

- UI/UX 개선
- 보고서 / 대시보드
- 데이터 내보내기

나중 — 보안 및 운영:

- 세분화된 권한
- 감사 로그
- 캐싱 / 성능 최적화
- 공격성 입력 방어 강화
- rate limit, 감사/탐지, 운영성 검증

---

## 2. 추상화는 반복 이후에만 한다

다음 경우에만 구조화를 고려한다.

- 동일 로직이 3회 이상 반복
- 동일 수정이 여러 파일에 걸쳐 반복
- 동일 규칙이 여러 곳에 흩어져 있음

막연한 미래 가능성으로 추상화하지 않는다.
세 줄의 유사한 코드가 섣부른 추상화보다 낫다.

---

## 3. 구조 변경 전 체크리스트

새 구조(레이어, 패턴, 추상화)를 추가하기 전에 확인한다.

1. 지금 반복되는 문제인가? (가설이 아닌 실제 반복)
2. 변경하지 않으면 수정 비용이 계속 증가하는가?
3. 과도한 일반화는 아닌가? (사용처가 1~2개면 불필요)
4. 자리만 만들어도 충분한가? (인터페이스, 빈 모듈, TODO 주석)

4개 중 하나라도 "아니오"면 구조 추가를 보류한다.
