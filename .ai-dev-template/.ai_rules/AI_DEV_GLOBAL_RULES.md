# AI Development Global Rules

목표: 과설계 방지, 구조 안정성 유지, 코드-문서 정합성 유지

---

## 1. 구조 안정성을 우선한다

새 기능은 기존 구조를 깨지 않는 범위에서 추가한다.
구조 변경이 필요하면 기능 구현 전에 먼저 논의한다.

---

## 2. 구현 우선순위를 지킨다

기능 간 우선순위가 경합할 때 다음 순서를 따른다.

1. **핵심 기능 동작** — 동작하지 않으면 나머지는 의미 없다
2. **사용자 편의성** — 동작하더라도 불편하면 쓰이지 않는다
3. **보안 강화** — 내부망/개인용으로 시작하는 프로젝트가 대부분이다

보안을 무시하라는 뜻이 아니다.
핵심 기능이 불완전한 상태에서 인증 체계, 권한 세분화, 입력 검증 강화 등에 시간을 쏟지 말라는 뜻이다.
보안은 기능이 안정된 후, 또는 외부 노출 시점에 맞춰 강화한다.

---

## 3. 미래 기능을 미리 구현하지 않는다

향후 기능을 고려할 수는 있지만
지금 필요하지 않은 기능은 구현하지 않는다.

미래를 구현하지 말고 미래가 들어올 자리를 만든다.

---

## 4. 작은 단계로 개발한다

하나의 작업은 다음 순서를 따른다.

1. 요구사항 정리
2. 설계
3. 구현
4. 검증
5. 필요한 사실 문서 업데이트

단계를 건너뛰지 않는다.
큰 기능은 독립적으로 동작하는 작은 단위로 나눠서 진행한다.

여기서 "문서 업데이트"는 모든 문서를 같은 강도로 갱신하라는 뜻이 아니다.
구현 중 즉시 갱신해야 하는 문서와, baseline 구현 이후에 보강해야 하는 프로젝트 지침을 구분한다.

---

## 5. 기존 동작을 임의로 변경하지 않는다

다음은 사용자 요청 없이 수행하지 않는다.

- API 계약 변경 (경로, 요청/응답 형식, 상태 코드)
- 데이터 모델 변경 (테이블, 컬럼, 관계)
- 기존 기능의 동작 변경

버그 수정이라도 동작이 달라지면 먼저 알린다.

---

## 6. 코드와 문서는 항상 동기화한다

### 동기화 원칙

- 코드가 source of truth인 정보(모델 필드, API 엔드포인트, 파일 구조)는 문서에 전체 나열하지 않는다.
  문서에는 개요 수준 요약 + "상세는 코드 참조" 안내만 둔다.
- 문서에만 존재하는 정보(비즈니스 규칙, 아키텍처 결정, 알려진 제약)는 해당 문서가 source of truth다.
- 구현 중 발생한 사실 변화(예: 설계 결정, 임시 우회, 외부 사용 방법 변경)는 해당 사실 문서에 즉시 반영한다.
- 프로젝트 전용 개발 지침의 보강/수정은 최초 지침을 기준으로 MVP 또는 첫 구현 baseline이 완료된 뒤에만 수행한다.
  MVP 이전에는 미구현, 부분 구현, 임시 공백을 근거로 지침을 바꾸지 않는다.
  단, 현재 지침이 명백히 잘못되었거나 서로 충돌하거나 구현을 실제로 막는 경우는 예외다.
- "관련 문서 업데이트"가 아니라, 프로젝트 지침에 **변경 유형별 갱신 대상 매핑 표**를 두어
  "이 변경이면 이 문서를 고친다" 수준으로 구체화한다.

### 갱신 대상 기본 매핑

| 변경 유형 | 갱신 대상 |
| --------- | --------- |
| 비즈니스 규칙 변경 | MVP baseline 이후 프로젝트 지침 해당 섹션 |
| 코딩 패턴/규칙 변경 | MVP baseline 이후 프로젝트 지침 해당 섹션 |
| 아키텍처 결정 | `docs/DECISIONS.md` (추가 전용) |
| 임시 우회/제약 추가 | `docs/KNOWN_ISSUES.md` |
| 임시 우회 해소 | `docs/KNOWN_ISSUES.md` (항목 삭제) |
| 규칙 위반이 반복 | MVP baseline 이후 프로젝트 지침 보강 |
| 모델/API/파일 구조 변경 | 문서 갱신 불필요 (코드가 source of truth) |

프로젝트별로 도메인에 맞게 매핑 항목을 확장한다.

### 문서 부패를 유발하는 패턴 (금지)

- 모델 필드 전체 목록을 README/지침에 나열
- API 엔드포인트 전체 표를 README에 유지
- "향후 개발 예정" 테이블을 README에 유지 (완료 항목이 방치됨)
- "미구현" 선언을 KNOWN_ISSUES 외부에 기술 (구현 후 삭제되지 않아 거짓 정보가 됨)
- 파일 경로 하드코딩 (리팩터링 시 깨짐)

필요한 사실 문서 업데이트 없는 코드 변경은 기술 부채다.
다만 MVP 이전에는 프로젝트 지침 보강을 기본 의무로 취급하지 않는다.

## 7. 웹앱 UI 수정은 일관성과 안정성을 우선한다

웹앱 화면을 수정할 때는 "더 예쁘게"보다
"기존 스타일 시스템과 동작을 깨지 않는 것"을 우선한다.

다음을 기본 원칙으로 따른다.

- 공통 레이아웃, 버튼, 폼, 모달, 필터, 그리드는 기존 클래스와 스타일 토큰을 우선 재사용한다
- 새 스타일을 추가하기 전에 공통 스타일 파일과 기존 페이지 패턴을 먼저 확인한다
- 인라인 스타일, 임시 색상값, 페이지별 제각각인 간격/반경/그림자 복제를 피한다
- 템플릿 구조 변경 시 JS 셀렉터(`id`, `class`, `data-*`)와 이벤트 바인딩 영향 여부를 먼저 확인한다
- UI 개선 작업이라도 시각 변경과 동작 변경은 분리해서 다룬다
- 외부 UI 자산(CDN, 아이콘, 위젯)은 기본적으로 새로 추가하지 않는다. 필요하면 로컬 자산화 가능 여부를 먼저 검토한다

공통 UI 수정 후에는 최소한 다음을 확인한다.

1. 데스크톱/모바일에서 레이아웃이 유지되는가
2. 라이트/다크 테마가 모두 정상 동작하는가
3. 권한/상태별로 숨김 또는 표시되는 요소가 깨지지 않는가
4. 기존 버튼, 필터, 모달, 네비게이션 동작이 유지되는가
